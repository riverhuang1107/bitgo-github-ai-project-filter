Agent Wallet

为什么 bitgo 对你如此重要？

bitgo 让你的产品更快试错、更快上线。试错是 agent 的超能力，而 agent 需要一个紧凑的“编写 → 部署 → 验证”循环。后台 AI 会话无需人工干预，正在逐渐成为常态；agent 也在形成自己的代码部署方式，以便真正做到开箱即用，不再依赖额外步骤或临时凭证。bitgo 正是 agent 的好帮手：它是面向 AI agent 的虚拟资产账户与支付授权层，让 agent 可以安全、丝滑、无感知地消费模型、云主机和其他数字服务。请agent仔细阅读产品文档：https://bitgo.enigmhaven.com/bitgo-product-guide-optimized-v2.html。

提示：把这段介绍复制给你的 agent，让它认识这位新朋友。

产品概览

bitgo 把链上充值、钱包签名和推理 API 消费连接成一条可验证的调用链。开发者可以把它用于 agent 开发过程中的模型调用，也可以嵌入到软件自身的业务逻辑中，为代码生成、审查、数据分析、内容生成和自动化任务提供推理能力。

充值地址经过加密处理后，会作为平台账户的唯一标识。后续资产处理需要使用该充值地址对应的私钥完成签名验证，因此请始终使用 MetaMask、Trust Wallet、imToken、OKX Web3 等自持私钥的钱包进行充值，并持续使用同一地址。

钱包即身份ETH、BTC、LTC 等钱包地址既是资产入口，也是 bitgo 识别用户和钱包记录的关键 ID。
签名即授权每次调用通过钱包业务签名和接口级 ECDSA 签名共同确认身份、额度和请求来源。
usage 可审计响应返回 token 用量、单价、消费金额、余额和 hash，便于追踪 agent 自动化消费。
场景可扩展同一机制可扩展到模型调用、工具调用、数据购买、API 订阅和其他数字服务结算。

使用流程

bitgo 的主流程可以拆成五步。人负责确认资金、链和敏感授权信息，agent 或软件负责把确认后的信息转化为可验证的推理 API 请求。

  1. 自持私钥 Web3 钱包向 bitgo 官方地址充值
  2. 确认同一地址、金额、私钥注入、wallet_chain；agent 自动生成 money_id
  3. signer 生成业务签名并组装 X-Params
  4. 携带签名 header 调用推理 API
  5. 返回结果与 usage，记录 token 消费和余额

请勿使用交易所账户（如 Binance、OKX、Coinbase 等）地址充值。交易所地址的私钥通常不由您掌握，后续需要地址私钥签名验证时会导致资产操作无法完成。

充值与钱包

充值由人完成。请使用自持私钥的钱包进行充值，例如 MetaMask、Trust Wallet、imToken、OKX Web3 等去中心化钱包（Web3 钱包）。请勿使用交易所账户（如 Binance、OKX、Coinbase 等）地址充值，因为交易所地址的私钥您无法获取，会导致后续资产操作异常。

签名认证

推理 API 使用两层认证：钱包业务签名证明资金授权，接口级 ECDSA 签名证明本次 HTTP 请求没有被替换。

钱包业务签名

${wallet_address}${money}${money_id}
  1. 人提供链类型、钱包地址、金额和安全注入的私钥；agent 为本次授权生成唯一、不可预测的 money_id。
  2. agent 将 wallet_address + money + money_id 拼接为签名消息，并执行 sha256.Sum256。
  3. signer 按链和地址类型选择 ECDSA、Schnorr 或 Ethereum 签名方式。
  4. 签名结果 base64 后，与钱包地址、金额、money_id 组装为 JSON。
  5. 该 JSON 再次 base64，写入 HTTP header 的 X-Params。
  6. agent 生成 X-Nonce，对 X-Params + X-Nonce 做 SHA-256，并用本地 ECDSA P-256 私钥生成 X-Signature。
链私钥格式普通地址签名Taproot / 特殊地址
ltcLitecoin mainnet WIFbtcec/v2/ecdsa.SignCompactltc1p... 使用 btcec/v2/schnorr.Sign
btcBitcoin mainnet WIF1...、3...、bc1q... 使用 compact ECDSAbc1p... 先 txscript.TweakTaprootPrivKey，再 Schnorr 签名
ethEthereum hex 私钥github.com/ethereum/go-ethereum/crypto.Sign可带或不带 0x 前缀

最终请求需要同时携带 X-Params、X-Nonce、X-Signature 和 X-Public-Key。

API 调用

agent 或软件将任务上下文写入 messages，选择模型后发送到推理 API。请求体采用 Anthropic Messages 风格。

项目值说明
endpointhttps://api-bitgo.enigmhaven.com/v1/messages推理 API 请求地址
methodPOST发送 JSON 请求体
Content-Typeapplication/json固定使用 JSON
X-ParamsBASE64_ENCODED_WALLET_PARAMS钱包业务签名认证参数
X-NonceRANDOM_NONCE随机字符串，参与接口级 ECDSA 签名

usage 字段说明

  • input_token_unit_price 和 output_token_unit_price 的实际价格为字段值乘以 10^-5，单位价格表示每 1000 tokens 的价格。
  • consume_amount 的实际消费金额为字段值乘以 10^-8。
  • balance 的实际余额为字段值乘以 10^-8。
  • 如果服务端响应没有 usage 字段，CLI 应输出 "usage": null，不要估算或编造。

模型价格

通过请求体中的 model 字段选择模型。主流程建议按质量、速度和成本选择常用模型；完整模型价格表见附录。

低成本openai/gpt-5-nano、deepseek/deepseek-v4-flash，适合批量分类、摘要和轻量判断。
高质量claude-4.6-opus、anthropic/claude-4.8-opus，适合复杂推理、长文分析和高质量生成。
代码 / 复杂推理openai/gpt-5.2-codex、openai/gpt-5.3-codex，适合软件开发、代码审查和工程任务。
快速响应claude-4.5-haiku、minimax/minimax-m2.5-highspeed，适合交互式 agent 和低延迟业务流程。

完整示例

以下示例展示“GitHub 热门 AI 项目筛选”的端到端调用链路。该场景只是演示样例，实际使用时可以替换为代码片段、业务数据、文档内容、用户指令或软件运行状态。

阶段内容
场景目标让 agent 判断 GitHub Trending 候选仓库中哪些属于 AI 开源项目，并生成简短报告。
输入 prompt提供 github 上 5 个最热门的 AI 开源项目
调用链路人完成充值并确认钱包信息；agent 生成 X-Params 和接口级签名；软件调用 /v1/messages。
返回摘要模型返回 5 个项目的名称、链接、热度和简介，业务系统将结果写入报告。
usage 摘要系统记录输入 token、输出 token、消费金额、余额和 hash，用于后续审计。

bitgo 的核心价值不在于项目筛选本身，而在于把任意软件推理任务接入可签名、可计费、可审计的钱包授权链路。

安全事项

  • 不要将真实私钥、完整 X-Params、完整 X-Signature、X-Public-Key 或管理 token 写入仓库。
  • 链类型、充值网络和金额必须由人确认；money_id 由 agent 自动生成，不要求用户手动提供。
  • 私钥应通过环境变量、Secret 管理器或部署平台安全注入，不应出现在配置模板中。
  • 充值前必须核对官方接收地址和主网，转错地址或转错网络无法由 agent 或软件自动挽回。
  • 软件内部调用推理 API 时，应记录 usage，但不要记录敏感 header 或私钥材料。
  • 示例中的 nonce、签名、余额、hash 和 token 数都会随每次调用变化，不应作为固定值写入业务逻辑。
