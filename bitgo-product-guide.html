Agent Reasoning Wallet

bitgo

bitgo 面向 agent 软件开发场景，为 agent 在开发软件过程中，以及 agent 开发出来的软件在自身业务流程中调用大模型推理时，提供链上钱包计费、签名授权和推理 token 消费能力。它不是单一项目筛选工具，而是让 agent 工作流和软件内部推理能力都能稳定接入模型服务的支付与认证层。

为 agent 和软件内置推理而设计

人负责确认链、地址、金额和授权信息；agent 负责创建钱包、生成签名请求并把推理能力接入开发流程或软件运行逻辑。

1. 产品亮点

Agent 时代的支付工具BitGo 面向 Agent 时代提供 去中心化的虚拟资产支付能力，让 AI Agent、自动化程序和开发者应用可以安全、可验证地使用加密货币钱包完成资源消费。它不依赖传统账户余额体系，而是以 钱包地址、链上身份和签名授权 为核心，把支付能力交还给用户和程序本身。
钱包即支付身份在 BitGo 中，ETH、BTC、LTC 等加密货币钱包不仅用于持有资产，也可以作为调用推理 API、模型服务、数据服务和其他数字资源的 授权凭证。每次请求都由 私钥签名 完成身份确认和消费授权，服务端无需托管用户私钥，也不依赖中心化账号密码体系。
安全透明的虚拟资产消费BitGo 让每一次资源消费都具备 清晰的来源、额度和记录。用户可以明确看到请求来自哪个钱包、消费了多少额度、对应哪个资金 ID，并通过完整 usage 信息追踪 token 用量、单价、消费金额和余额变化，从而让 Agent 的自动化支付行为保持 可审计、可验证。
从 Token 消费到 Agent 经济消费 token 只是 BitGo 的起点。同一套 钱包签名和支付机制 可以继续扩展到模型调用、工具调用、数据购买、API 订阅、算力租赁、自动化交易、任务赏金和跨平台数字服务结算，成为连接钱包、资产、服务与 Agent 自动化消费行为的 基础支付层。

3. 链上地址与钱包创建

bitgo 以链上地址作为用户唯一识别 ID。一个链上地址可以对应多个钱包，用来支持不同 agent、不同软件项目、不同运行环境或不同业务模块的推理 token 消费隔离。

人提供的信息链类型 ltc、btc 或 eth，钱包地址，充值金额或面额，money_id，以及通过安全方式注入的私钥。
agent 完成的动作创建或配置钱包记录，生成签名请求，测试推理 API 连通性，并把模型调用接入 agent 开发流程或软件内部业务逻辑。

链类型、金额和授权信息必须由人明确提供。agent 不应根据地址外观擅自推断业务金额，也不应把真实私钥写入仓库、文档、日志或示例代码。

6. 多链钱包生成与连通性测试

加密货币相关签名由 Go signer 统一完成。agent 或软件运行时调用 signer 组装 X-Params，再使用本地 ECDSA P-256 私钥为 X-Params + X-Nonce 生成接口级签名，最后发送完整推理 API 请求。不同链的钱包生成和签名方式如下：

LTC
生成 Litecoin mainnet 钱包，导出地址和 WIF 私钥，配置 wallet_chain = "ltc"。普通地址使用 compact ECDSA；Taproot 地址 ltc1p... 使用 Schnorr 签名。LTC 已通过真实请求验证，但链类型仍必须由人明确提供。
BTC
生成 Bitcoin mainnet 钱包，导出地址和 WIF 私钥，配置 wallet_chain = "btc"。普通地址如 1...、3...、bc1q... 使用 compact ECDSA；Taproot 地址 bc1p... 使用 tweak 后的 Schnorr 签名。
ETH
生成 Ethereum 钱包，导出 0x... 地址和 hex 私钥，配置 wallet_chain = "eth"，由 signer 使用 Ethereum 签名方式生成授权参数。 
curl -X POST "https://api-token-enigmhaven.expvent.com.cn:1111/v1/messages" \
  -H "Content-Type: application/json" \
  -H "X-Params: BASE64_ENCODED_WALLET_PARAMS" \
  -H "X-Nonce: RANDOM_NONCE" \
  -H "X-Signature: ECDSA_SIGNATURE_HEX" \
  -H "X-Public-Key: ECDSA_PUBLIC_KEY_HEX" \
  --data-raw '{
    "model": "claude-4.6-opus",
    "max_tokens": 1024,
    "stream": false,
    "messages": [
      {
        "role": "user",
        "content": "你好"
      }
    ]

连通性测试成功后，说明钱包签名、X-Params 组装、接口级 ECDSA 签名、模型选择和推理 API 访问链路已经打通。agent 可把同一套链路复用到代码生成、审查、数据分析、业务自动化或软件内部推理任务。

7. 钱包签名调用推理 API

推理 API 使用 X-Params 钱包业务签名，并叠加 X-Nonce、X-Signature、X-Public-Key 接口级 ECDSA 签名。钱包签名消息由钱包地址、金额和金额 ID 直接拼接得到：

${wallet_address}${money}${money_id}

X-Params 签名过程

  1. 人提供 wallet_chain、wallet_address、money、money_id 和安全注入的私钥。
  2. agent 将 wallet_address + money + money_id 拼接为签名消息。
  3. signer 对签名消息执行 sha256.Sum256，得到 32-byte digest。
  4. ltc 解码 Litecoin mainnet WIF 私钥后按地址类型选择签名：普通地址调用 btcec/v2/ecdsa.SignCompact，Taproot 地址 ltc1p... 调用 btcec/v2/schnorr.Sign。
  5. btc 解码 Bitcoin mainnet WIF 私钥；普通地址如 1...、3...、bc1q... 使用 btcec/v2/ecdsa.SignCompact；Taproot 地址 bc1p... 先调用 txscript.TweakTaprootPrivKey(*privateKey, []byte{}) 调整内部私钥，再使用 btcec/v2/schnorr.Sign。
  6. eth 使用 Ethereum hex 私钥，可带或不带 0x，并调用 github.com/ethereum/go-ethereum/crypto.Sign。
  7. 签名结果 base64 编码后，与 wallet_address、money、money_id 组装为 JSON。
  8. 将该 JSON 字符串再次 base64 编码，最终写入 HTTP header 的 X-Params。
  9. 生成随机 X-Nonce，将 X-Params + X-Nonce 直接拼接后执行 SHA-256。
  10. 使用本地 ECDSA P-256 私钥生成 ASN.1/DER 签名 hex，写入 X-Signature，并把本地 ECDSA 公钥 DER hex 写入 X-Public-Key。
字段说明
wallet_address链上地址，也是 bitgo 识别用户和钱包的关键 ID
money人提供的金额或面额，不由 agent 猜测
money_id金额记录或钱包记录 ID，用于服务端识别
signature对 wallet_address + money + money_id 签名后的 base64 字符串

签名字段组装为 JSON 后再次 base64，放入请求 header，并为接口级请求生成 ECDSA 签名：

{
  "wallet_address": "YOUR_WALLET_ADDRESS",
  "money": "YOUR_WALLET_MONEY",
  "money_id": "YOUR_MONEY_ID",
  "signature": "BASE64_SIGNATURE"
}
Content-Type: application/json
X-Params: BASE64_ENCODED_WALLET_PARAMS
X-Nonce: RANDOM_NONCE
X-Signature: ECDSA_SIGNATURE_HEX
X-Public-Key: ECDSA_PUBLIC_KEY_HEX

推理请求体使用 Anthropic Messages 风格。agent 可以把代码上下文、数据样本、业务指令或软件内部任务写入 messages，由 bitgo 钱包签名授权后调用模型。

8. 完整示例：GitHub 热门 AI 项目筛选

该示例展示的是一条完整 bitgo 调用链路，不代表 bitgo 只能用于项目筛选。任何 agent 开发的软件，只要内部需要大模型推理，都可以替换输入数据、提示词和输出结构。

GitHub 热门 AI 项目筛选只是本文的演示样例，用来展示充值、钱包创建、签名调用、推理消费和结构化结果返回的完整链路；bitgo 的使用范围不局限于项目筛选。

  1. 人完成链上充值，并确认资金已入账。
  2. 人提供链类型、钱包地址、金额、money_id 和安全注入的私钥。
  3. agent 创建钱包配置，生成钱包参数与接口级签名 header。
  4. agent 开发的软件调用推理 API，对 GitHub Trending 候选仓库执行 AI 相关性判断。
  5. 服务端返回结构化 JSON 与 usage，系统生成报告并记录 token 消费。

usage 字段说明

每次外部推理 API 请求完成后，服务端会在响应中返回完整 usage JSON。上面的完整响应示例已经包含 usage 字段，因此不再单独重复展示一份 usage JSON。

usage 金额字段为服务端返回的整数缩放值，可按以下比例换算为实际金额：

  • input_token_unit_price 和 output_token_unit_price 的实际价格为字段值乘以 10^-5，且单位价格表示每 1000 tokens 的价格。
  • consume_amount 的实际消费金额为字段值乘以 10^-8。
  • balance 的实际余额为字段值乘以 10^-8。

如果服务端响应没有 usage 字段，CLI 会输出 "usage": null，不会估算或编造。

9. 安全事项

  • 不要将真实私钥、完整 X-Params、完整 X-Signature、X-Public-Key 或管理 token 写入仓库。
  • 链类型、充值网络、金额和 money_id 必须由人确认后交给 agent 使用。
  • 私钥应通过环境变量、Secret 管理器或部署平台安全注入，不应出现在配置模板中。
  • 充值前必须核对官方接收地址和主网，转错地址或转错网络无法由 agent 或软件自动挽回。
  • 软件内部调用推理 API 时，应记录 usage，但不要记录敏感 header 或私钥材料。
