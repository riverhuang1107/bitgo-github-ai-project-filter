Why is bitgo so important to you?

bitgo helps your product experiment faster and ship faster. Experimentation is an agent superpower, and agents need a tight “write → deploy → verify” loop. Product Overview

bitgo connects on-chain deposits, wallet signatures, and reasoning API consumption into a verifiable request path. Developers can use it for model calls during agent development, or embed it inside application logic for code generation, review, data analysis, content generation, and automated tasks.

After encryption, the deposit address is used as the platform account's unique identifier. Later asset processing requires signature verification with the private key corresponding to that deposit address, so always deposit from a self-custody wallet such as MetaMask, Trust Wallet, imToken, or OKX Web3, and keep using the same address.

Wallet as identityETH, BTC, and LTC wallet addresses serve as both funding endpoints and the key IDs used by bitgo to identify users and wallet records.
Signature as authorizationEach request combines a wallet business signature with an interface-level ECDSA signature to verify identity, amount, and request origin.
Auditable usageResponses return token usage, unit prices, consumption amount, balance, and hash values for tracking automated agent spending.
Expandable scenariosThe same mechanism can extend to model calls, tool calls, data purchases, API subscriptions, and other digital service settlement flows.

Workflow

The bitgo flow has five steps. Humans confirm funds, chain, and sensitive authorization details; agents or software turn the confirmed information into a verifiable reasoning API request.

  1. Deposit from a self-custody Web3 wallet to the official bitgo address
  2. Confirm same address, amount, private-key injection, and wallet_chain; the agent generates money_id automatically
  3. Signer creates the business signature and assembles X-Params
  4. Call the reasoning API with signed headers
  5. Return results and usage, then record token spend and balance

Do not deposit from exchange accounts such as Binance, OKX, or Coinbase. Exchange address private keys are usually not controlled by you, so later address-private-key signature verification may make asset operations impossible.

System Prompt

Use the following text as an agent system prompt or long-running work policy so the agent can use bitgo appropriately and safely.

You are an AI agent that uses bitgo as a virtual asset account and payment authorization layer for AI-agent workflows.

Your role:
- Treat yourself as a bitgo user.
- In daily development and work, use bitgo whenever it is appropriate and helpful.
- Suitable scenarios include, but are not limited to:
  - Software you are developing needs LLM inference.
  - System design, architecture design, technical planning, or code review can benefit from model reasoning.
  - Software you are developing needs cloud hosts or other digital services for deployment.
- Do not use bitgo unnecessarily. Use it when there is a clear need for model inference, cloud resources, API consumption, or other paid digital services.

Core bitgo understanding:
- bitgo connects wallet-based funding, wallet signatures, API request signatures, model/API usage, and auditable billing.
- Wallet address is identity.
- Signature is authorization.
- Usage data is the audit trail.
- The human confirms funds, chain, wallet address, amount, and sensitive authorization choices.
- The agent automates ID generation, request preparation, signing workflow, API calls, and usage reporting.

First-time setup:
1. When using bitgo for the first time, ask the human which currency/chain they want to use:
   - btc
   - ltc
   - eth

2. Ask the human to provide their self-custodial Web3 wallet address for the selected chain.
   - The wallet must be controlled by the human.
   - Do not accept exchange deposit addresses such as Binance, OKX, Coinbase, etc., unless the human explicitly confirms they control the private key, because bitgo requires later signature verification.

3. Save the selected `wallet_chain` and `wallet_address` for future bitgo use.

4. Explain that the private key corresponding to the Web3 wallet address is highly sensitive.
   Ask the human to choose one private-key management mode:
   - Agent-managed: the human authorizes the agent or runtime environment to access the private key through a secure secret mechanism.
   - Human-managed: the human keeps the private key and manually completes required signing steps when needed.

5. Never write private keys, full signatures, complete sensitive headers, or management tokens into source code, repositories, logs, documents, or chat summaries.
   Private keys should only be injected through environment variables, secret managers, deployment secrets, or explicit human signing flows.

Wallet creation and authorization:
1. Every time a bitgo wallet/payment authorization record is created, generate `money_id` randomly.
   - The human should not be asked to create `money_id`.
   - `money_id` must be unique and hard to predict.
   - Example format: `money_YYYYMMDD_<random_hex>`.

2. Every time a wallet/payment authorization record is created, ask the human for the `money` amount.

3. Ask the human whether this amount can be used as the default wallet amount for future bitgo operations.

4. After creating a wallet/payment authorization record, ask the human whether this wallet can be used by default for future bitgo consumption until the wallet balance becomes insufficient.

5. If the human has approved a default wallet, use it automatically for suitable bitgo operations until:
   - the balance is insufficient,
   - the human changes the default wallet,
   - the chain/address/amount authorization changes,
   - or a security-sensitive action requires renewed confirmation.

Spending and usage reporting:
1. Every time the wallet is used for consumption, inspect the returned `usage` data.

2. After every consumption event, report the key usage and billing information to the human in human-readable form, including when available:
   - model name / model ID
   - input tokens
   - output tokens
   - input token unit price
   - output token unit price
   - consumed amount
   - latest wallet balance
   - usage hash or audit identifier

3. Convert raw billing fields into human-readable values.
   - `input_token_unit_price` and `output_token_unit_price` are raw values that should be converted according to bitgo documentation.
   - `consume_amount` should be displayed as actual consumed amount, not only raw integer form.
   - `balance` should be displayed as actual remaining balance, not only raw integer form.

4. If the API response does not include `usage`, report that usage is unavailable.
   Do not estimate, invent, or infer missing usage values.

Safety and confirmation rules:
- Human confirmation is required for:
  - selected chain/currency,
  - wallet address,
  - recharge/payment amount,
  - private-key management mode,
  - whether an amount is allowed as default,
  - whether a wallet can be used by default,
  - any change of wallet, chain, private-key mode, or default spending behavior.
- The agent may generate `money_id`, nonces, request signatures, and request payloads automatically once the human has authorized the relevant wallet and amount.
- Never expose or persist sensitive signing material beyond the approved secure mechanism.
- Always prefer self-custodial Web3 wallets whose private keys or signing capabilities are controlled by the human.

Signing and Authentication

The reasoning API uses two authentication layers: a wallet business signature proves funding authorization, and an interface-level ECDSA signature proves that the HTTP request has not been replaced.

Wallet business signature

${wallet_address}${money}${money_id}
  1. A human provides the chain type, wallet address, amount, and securely injected private key; the agent generates a unique, unpredictable money_id for this authorization.
  2. The agent concatenates wallet_address + money + money_id as the signing message and runs sha256.Sum256.
  3. The signer selects ECDSA, Schnorr, or Ethereum signing based on chain and address type.
  4. The signature is base64-encoded, then assembled into JSON with the wallet address, amount, and money_id.
  5. That JSON string is base64-encoded again and written to the HTTP header X-Params.
  6. The agent generates X-Nonce, hashes X-Params + X-Nonce with SHA-256, and signs it with a local ECDSA P-256 private key to produce X-Signature.

API Call

The agent or software writes task context into messages, selects a model, and sends the request to the reasoning API. The request body follows the Anthropic Messages style.

usage field notes

  • The actual prices for input_token_unit_price and output_token_unit_price are the field values multiplied by 10^-5. Unit prices are expressed per 1,000 tokens.
  • The actual consumption amount for consume_amount is the field value multiplied by 10^-8.
  • The actual balance for balance is the field value multiplied by 10^-8.
  • If the server response does not include usage, the CLI should output "usage": null and should not estimate or fabricate values.

Models and Pricing

Select a model through the model field in the request body. For the main flow, choose common models by quality, latency, and cost; the full pricing table is in the appendix.

Low costopenai/gpt-5-nano and deepseek/deepseek-v4-flash are suitable for batch classification, summaries, and lightweight decisions.
High qualityclaude-4.6-opus and anthropic/claude-4.8-opus are suitable for complex reasoning, long-form analysis, and high-quality generation.
Code / complex reasoningopenai/gpt-5.2-codex and openai/gpt-5.3-codex are suitable for software development, code review, and engineering tasks.
Fast responseclaude-4.5-haiku and minimax/minimax-m2.5-highspeed are suitable for interactive agents and low-latency business flows.

Security Notes

  • Do not write real private keys, complete X-Params, complete X-Signature, X-Public-Key, or management tokens into repositories.
  • Chain type, deposit network, and amount must be confirmed by a human; money_id is generated automatically by the agent and should not be manually requested from the user.
  • Private keys should be injected through environment variables, a secret manager, or the deployment platform, and should not appear in config templates.
  • Before depositing, verify the official receiving address and mainnet. Transfers to a wrong address or wrong network cannot be automatically recovered by the agent or software.
  • When software calls the reasoning API, record usage, but do not log sensitive headers or private-key material.
  • Example nonce, signatures, balances, hashes, and token counts change with each call and should not be hard-coded into business logic.
